NeoQCR Infrastructure as Code (IaC)This repository contains the Terraform configurations for deploying and managing the NeoQCR Azure infrastructure, specifically focusing on the private Azure Application Gateway (WAF_v2) and its associated networking components.🏗️ Architecture & Directory StructureThis repository uses a modular approach to strictly separate reusable infrastructure logic from environment-specific configurations.neoqcr-infrastructure/
├── modules/
│   └── application_gateway/       # Reusable Terraform module
│       ├── main.tf                # Core App Gateway and WAF resources
│       ├── variables.tf           # Module inputs
│       └── outputs.tf             # Module outputs
├── environments/
│   ├── dev/                       # Development Environment
│   │   ├── main.tf                # Invokes the module with dev variables
│   │   ├── providers.tf           # Connects to the dev remote state
│   │   └── terraform.tfvars       # Dev-specific IP configurations
│   └── prod/                      # Production Environment
│       ├── main.tf
│       ├── providers.tf           # Connects to the prod remote state
│       └── terraform.tfvars
└── .github/workflows/
    └── terraform.yml              # GitHub Actions CI/CD Pipeline
🛡️ Enterprise IaC Principles EnforcedThis repository strictly adheres to the following 8 infrastructure principles:Manipulate state only through TF commands: Manual changes in the Azure Portal will be detected as drift and overwritten by the pipeline.Remote state: State is never stored locally. It is stored securely in an Azure Blob Storage container.State locking: Azure Blob Storage automatically leases the state file during execution, preventing parallel pipeline runs from corrupting the infrastructure.Backup state file: Blob Versioning is enabled on the storage account, keeping a historical backup of every state change.Use 1 state per environment: The dev and prod environments use entirely separate .tfstate files, ensuring mathematical isolation.Host TF in a git repository: All infrastructure changes are version-controlled and auditable here.CI for TF code: Every Pull Request automatically triggers terraform fmt, validate, and plan to catch errors before code merges.Execute TF only in an automated build: Engineers do not have local access to the deployment credentials. Deployments must pass through the GitHub Actions pipeline.⚙️ PrerequisitesTo allow the GitHub Actions pipeline to deploy to Azure, the following Repository Secrets must be configured in GitHub (Settings -> Secrets and variables -> Actions):AZURE_CLIENT_ID: The App ID of the Service Principal.AZURE_CLIENT_SECRET: The password for the Service Principal.AZURE_TENANT_ID: Your Azure Active Directory Tenant ID.AZURE_SUBSCRIPTION_ID: Your Azure Subscription ID.Note: The Service Principal must have Contributor rights on the target Azure Subscription.🚀 Deployment WorkflowLocal executions of terraform apply are strictly forbidden and will fail due to lack of credentials. All changes must follow this GitOps workflow:1. Make Changes (Continuous Integration)Create a new branch off main.Update the .tf or .tfvars files in the respective environments/ folder.Open a Pull Request (PR) against the main branch.Automated CI Check: GitHub Actions will automatically run the terraform-ci-plan job. Review the Plan output in the PR checks to see exactly what will be created, modified, or destroyed.2. Deploy (Continuous Deployment)Once the PR is approved and the CI checks pass, merge the PR into main.Automated CD Trigger: GitHub Actions will automatically trigger the terraform-cd-apply job.The pipeline will execute terraform apply -auto-approve using the secure credentials and update the Azure infrastructure.🔧 Bootstrapping a New State StorageIf you are setting this up in a brand new Azure Tenant, you must create the remote state storage account before the pipeline can run. Run the provided setup_backend.sh script locally once to provision the resource group, storage account, and container with blob versioning enabled. Update providers.tf with the generated storage account name.
